# Plugin Release Followup

这个 skill 只负责一件事：当第一方插件发生改动时，确保“先功能提交，后发布提交”的收尾流程被完整执行。

不要用这个 skill 做业务开发、代码修改方案设计或普通仓库提交。它只服务插件改动后的发布补动作。

## 何时使用

在以下任一情况，必须使用本 skill：

- 本次工作改动涉及 `packages/<plugin>/`
- 你准备提交插件相关改动
- 你已经做完插件功能提交，但还没有补做发布提交和 tag
- 一次改动涉及多个第一方插件，需要逐个发布

如果当前改动不涉及第一方插件目录，这个 skill 应立即退出，不接管后续流程。

## 工作流程

### 1. 提交前检测

先检查本次工作影响了哪些第一方插件。

做法：

- 查看工作区 diff、暂存区或最近准备提交的文件范围
- 从中提取受影响的 `packages/<plugin>/` 目录
- 形成受影响插件列表

要求：

- 如果没有插件目录改动，退出本 skill
- 如果命中了多个插件，全部记录下来
- 不允许只挑一个插件处理

### 2. 功能提交

先完成正常功能提交。

要求：

- 功能提交只承载业务改动
- 不要在这个提交里手工混入版本 bump、registry 更新或 tag 创建
- 功能提交完成后，不能把任务视为结束

### 3. 提交后补发布

功能提交完成后，按受影响插件列表逐个执行：

```bash
./scripts/release/ak-release.sh <plugin-id> <bump-type>
```

每个插件都必须得到自己的：

- 发布提交
- 插件级 tag

不要尝试跳过脚本、手写版本号或手动模拟发布步骤。发布入口固定就是 `./scripts/release/ak-release.sh`。

## 多插件规则

如果一次功能提交影响多个第一方插件：

- 允许一次功能提交后产生多个连续发布提交
- 所有受影响插件都必须发布
- 固定按 `plugin_id` 字典序逐个执行发布

使用字典序是为了让行为稳定、可预测，也方便 review 和回溯。

## bump 类型规则

不要随意猜测版本类型。

固定规则：

- 如果当前任务上下文已经明确版本语义，就使用对应的 `patch` / `minor` / `major`
- 如果上下文没有明确说明，默认使用 `patch`
- 不允许在没有明确依据时把默认值提升为 `minor` 或 `major`

## 禁止事项

以下行为都不允许：

- 只做功能提交，不补后续发布提交
- 只发布多个受影响插件中的一部分
- 不通过 `./scripts/release/ak-release.sh` 而改走手工发布
- 在没有依据时擅自选择 `minor` 或 `major`
- 把这个 skill 用作普通开发流程

## 完成标准

只有同时满足以下条件，才能认为插件改动收尾完成：

- 受影响插件列表已经识别完成
- 功能提交已经完成
- 每个受影响插件都已经调用 `./scripts/release/ak-release.sh`
- 多插件场景下，发布顺序符合 `plugin_id` 字典序
- 发布补动作没有遗漏任何插件
